# Dashboard Sync — the Claude half of the CWDB HQ two-way loop

The CWDB HQ dashboard (operations/dashboard/) writes an append-only event log
to Supabase `dashboard_events` every time Jim acts on it (completes/defers/
declines a task, adopts a counsel move, queues work for Claude, records a
decision). This skill folds those events into Claude's world WITH JUDGMENT —
the dashboard never edits memory files itself.

## Steps

### 1. Read unprocessed events

```sql
SELECT event_id, event_type, task_id, payload, actor, created_at
FROM dashboard_events WHERE processed_at IS NULL ORDER BY event_id;
```

Use the Supabase MCP tool (project `iabiwsbmnbxmkjvkgfhg`). Join task context
when useful: `SELECT * FROM dashboard_tasks WHERE task_id IN (...)`.

### 2. Process each event by type

- **queued_for_claude** — this is work Jim explicitly sent. The full prompt is
  in `payload.prompt`. Execute it now (respecting the suggested agent) unless
  it is clearly a large project, in which case plan it and tell Jim what you
  propose. This is the highest-priority event type.
- **task_status_change / task_created / task_edited** — update your picture of
  the work. Only write memory when the event carries durable signal:
  - a DECLINED audit fix = a decision worth recording (why did Jim decline?
    if the reason in `payload.reason`/notes is unclear, note it as open);
  - a completed P0 (license, insurance, Quinn path) = update the relevant
    memory file / MEMORY.md line if it changes standing facts;
  - routine completions = no memory write, just awareness.
- **decision** — Jim recorded a decision on the dashboard (e.g. tax set-aside).
  Record in the session note; add to MEMORY.md ONLY if durable and non-obvious.
- **counsel_convened / refresh_run / sent_to_terminal** — operational
  breadcrumbs. No memory writes; skip straight to marking processed.
- **session_summary** — emitted by /session-end from another session; context
  only.

### 3. Update surfaces (judgment, not transcription)

- Session note (`_vault/sessions/<today>-NNN.md`): add one line per meaningful
  event under Work Done or Decisions Made.
- MEMORY.md / memory files: only durable standing facts, per the memory
  discipline in MEMORY.md.
- Agent memory (`~/.claude/agent-memory/<agent>/`): only platform quirks or
  constraints an agent will need again.
- Board mirrors are regenerated by the dashboard itself — do NOT hand-edit
  `_vault/board/*.md`.

### 4. Mark events processed

After handling each event (including skips):

```sql
UPDATE dashboard_events SET processed_at = now() WHERE event_id IN (...);
```

Never mark an event processed if you deferred its work without telling Jim.

### 5. Close the loop back to the dashboard

If you executed a queued task, update its row so the dashboard shows it:

```sql
UPDATE dashboard_tasks SET status='done', completed_at=now(),
  notes=COALESCE(notes || E'\n', '') || '<one-line summary>',
  updated_at=now() WHERE task_id = <id>;
INSERT INTO dashboard_events (event_type, task_id, payload, actor, processed_at)
VALUES ('task_status_change', <id>,
        '{"new_status":"done","by":"dashboard-sync"}', 'claude', now());
```

(Insert your own event pre-processed — you are its processor.)

## Report

End with a short summary: N events processed, what was executed, what was
recorded to memory, anything left open.
