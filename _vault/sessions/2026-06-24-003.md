# Session 2026-06-24-003

## Context
Jim reported the Mission Control dashboard at `http://127.0.0.1:7717` returning ERR_CONNECTION_REFUSED (screenshot). Goal: diagnose why it went down and restore it, then decide whether to also fix the underlying design flaw so it stops silently dying.

## Work Done
- **Root-caused the outage (read-only diagnosis).** Port 7717 had no TCP listener and no URL ACL; no `dashboard-server.ps1` process existed (the only WinPS 5.1 processes were WindowsTerminal children); yet `\CWDB\CWDB-Dashboard` reported State=Running / LastResult=`0x41301`. Diagnosis: the server process died (likely the `HttpListener.GetContext()` accept loop threw on a sleep/resume, outside the try/catch, under `ErrorActionPreference=Stop`), Task Scheduler wedged in a ghost "Running" state, and the 5-min babysitter (`MultipleInstances=IgnoreNew`) treated the ghost as "already up" and suppressed every relaunch. No logs because the task launched hidden with no stdout/stderr redirect. Confirmed it was not a code regression (last dashboard commit was 2026-06-11).
- **Immediate recovery.** `Stop-ScheduledTask` (cleared the ghost) + `Start-ScheduledTask` restored the dashboard within seconds (port 7717 Listen, HTTP 200). This also ruled out a startup/env failure: the wedge was the sole cause.
- **Hardened the keep-alive (restart + harden plan, approved).**
  - New `dashboard-babysitter.ps1`: short-lived task action that probes the actual port each tick and relaunches `dashboard-server.ps1` (detached, output to `_logs/`) only if down. Task returns to Ready every tick, so it cannot wedge. Liveness judged by the port, not Task Scheduler bookkeeping.
  - `install-dashboard-task.ps1`: task action now runs the babysitter.
  - `dashboard-server.ps1`: added a top-level `trap` (logs + exit 1) and a guarded `GetContext()` accept loop that restarts the listener in place on a sleep/resume `HttpListenerException` instead of dying silently.
  - `.gitignore`: added `operations/control-plane/dashboard/_logs/`.
- **Verified end-to-end.** Self-test passed on an alt port; redeployed via the installer (clean rotation); port reopened; **kill-test**: killed server PID 12596, the babysitter self-healed in ~24s, the task returned to **Ready** (no ghost wedge), and the prior run's output was archived to `.prev`. Final probe HTTP 200, `/api/state` returned live data (7 tasks, 1 pending approval).
