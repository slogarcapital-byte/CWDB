## Decisions Made

1. **Helper-view pattern over per-view WHERE clauses.** `v_clean_leads` + `v_clean_traffic` give a single source of truth for the test-exclusion list. Future test sources only need to be added in one place. Trade-off: one extra view-lookup hop per query, negligible at warehouse scale.

2. **Edit `001_views.sql` in place + use migration 006 for the deploy record.** `CREATE OR REPLACE VIEW` is idempotent, so re-running the file is safe and the canonical source stays single-file. The DB-side migration ledger (`apply_migration` name = `006_phase_e_test_exclusion_and_reconciliation_views`) captures the change-event for audit.

3. **Diagnosis columns on the two reconciliation views.** `v_meta_attribution_gap.diagnosis` returns `pixel_routing_broken` / `agree` / etc rather than just numbers. `v_thankyou_reconciliation.match_status` returns `lead_without_thankyou_event` / `thankyou_without_lead` / `match` / `partial_match`. Makes the views self-interpreting without SQL knowledge.

4. **Defer backfill scripts.** Phase E does not need them. The original "backfill" requirement was max-history pulls, and raw_hubspot_snapshot already reaches back to 2026-03-28. The real backfill gaps (Gmail-recoverable pre-relay leads, fact_leads loader selection) are next-session workstreams.

5. **Phase E declared shipped** after 8/8 verification PASS. Five follow-on findings (UTM black hole, pre-relay lost leads, cron silently broken, raw-to-fact gap, fact_bids quality) flagged for memory writes and next session, but they are Phase-E-adjacent, not Phase-E-blockers.
