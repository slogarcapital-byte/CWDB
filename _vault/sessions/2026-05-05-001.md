## Work Done

- **Walked through current state of Webflow ↔ HubSpot integration** — surfaced what was live (form → email only), parked (Make scenario, HubSpot OAuth in Make), missing (no pipeline, no wiring, no GCLID capture, no fan-out), and the manual workaround (email → Jim reads → Jim SMS Ben/John).

Updated `01-pipeline-stages.csv` and `hubspot-lead-pipeline.json` to match canonical 9-stage version.
- **Corrected major spec error mid-session.** Original `02-deal-properties.csv` had 10 form-fillable Deal properties, which is architecturally impossible in HubSpot (forms can only populate Contact, Company, Ticket — not Deal). Caught by Jim when HubSpot's form builder showed only 3 property categories. Deleted the wrong file, created `02-contact-properties.csv` (11 form-fillable Contact properties) and `03-deal-properties.csv` (8 workflow-managed Deal lifecycle properties). Updated README and JSON spec to reflect corrected architecture: form populates Contact, workflow creates Deal and associates to Contact.
- **Diagnosed HubSpot Webflow App "Could not get or set child form element ID" error.** Root cause: form is inside a Webflow component instance (`quote-form-wrap` is a multi-step component with hidden steps). The App tries to mutate child element IDs but Webflow blocks writes to component-instance children. Recommended Jim open the component definition. Error persisted; concluded the App is incompatible with the 2026-04-28 form rebuild's multi-step + JS-controlled architecture. Recommended abandon Path A entirely.
- **Pivoted to Phase 0 gate Plan A (Non-HubSpot Forms tracking script).** Walked Jim through GTM tag install + HubSpot tracking-code setup + Non-HubSpot Forms collection enablement. Phase 0 PASSED — test contact landed cleanly with built-in fields (firstname, email, phone, address, zip).
- **Identified Plan A's weakness during Phase 0 testing.** HubSpot's auto-detection inconsistently used field `name` for some fields (`timeline`) and label-derived snake_case for others (`estimated_budget`, `anything_else_we_should_know_`, `by_submitting_i_agree_...`). Jim asked the smart operator question: "can't I just create a webhook with HubSpot?" — which was the right instinct. Explained the Webflow native webhook send target → HubSpot Forms API has a payload shape mismatch, and HubSpot Starter Customer Platform doesn't include Operations Hub Pro's webhook trigger. Recommended Plan B: Forms API direct fetch via JS interceptor on the page.
- **Created HubSpot form schema** (form GUID `bb473d64-06b1-4311-8e02-7c70d605b79b`, never embedded) to give the Forms API endpoint a target.

Defensive against both `name`-attr variants for renamed fields (`owns_property`/`ownership`, `project_type`/`project-type`).

## Decisions Made

1. **Pipeline stages expanded from 6 to 9.** Jim's process knowledge layered on. Bid lifecycle now modeled as three distinct contractor activities (Creating → Delivered → Accepted) instead of one. Surfaces contractor-side bottlenecks separately from homeowner-side ghosting.
2. **`Expired Bid - No Response` kept as OPEN stage (no Closed flag).** Preserves re-engagement optionality. Initial win prob 50% is Jim's gut estimate; flagged for revisit after first 10 leads land actual conversion data.
3. **Form-fillable lead detail moved from Deal to Contact properties.** HubSpot architectural constraint: forms cannot populate Deal properties. Lead detail belongs to the *person*, not the *deal*. Workflow tomorrow creates the Deal and associates to the Contact; reports join via association.
4. **Webflow App abandoned for HubSpot integration.** Component-lock + multi-step form structure makes it incompatible with the 2026-04-28 form rebuild. Permanent decision; do not retry.
5. **Forms API direct fetch chosen over Non-HubSpot Forms tracking script.** Plan A worked at the architecture level but had inconsistent field-name auto-detection. Plan B (explicit JS interceptor with field-name map) is deterministic and survives form redesigns cleanly. ~30 min implementation, 50 lines of JS, replaces 3 planned integrations (Make + Twilio + custom serverless).
6. **Slug values as internal names + friendly labels.** Webflow form sends `under-10k`; HubSpot Contact property internal name = `under-10k`, label = "Under $10,000". Single source of truth for option values, no script-side translation needed, future-proofs workflow branch logic.
