# Session 2026-04-27-002

## Context

Emergency response: customer-reported `/get-a-quote` Submit button broken on mobile, reproduced by Jim on iPhone 11 Safari AND iPhone Chrome (which uses WebKit). Three rounds of misdirected fixes before correct root cause was found, then full surgical rebuild of the form's architecture, scripts, and field structure. Production-verified by Jim's real-device test at end of session.

Started as a one-line CSS bug. Ended as the form-architecture cleanup that should have happened on day-1 launch — runtime field injection killed, 15 stacked legacy scripts consolidated to 8, name+email promoted to first-class Webflow fields, project info colocated on Step 3.

## Work Done

**Round 1 — false negative.** Initial Explore + web-dev diagnostic. Playwright Chromium test passed end-to-end. Concluded Jim was testing in raw Designer canvas (where Submit is intentionally inert). WRONG — Jim was on production.

**Round 2 — wrong root cause.** Pivoted to "iOS Safari + flex display on form-control submit" hypothesis. web-dev registered `cwdb_ios_submit_fix-1.0.0` (CSS override forcing `display: inline-block`). Insufficient.

**Round 3 — claimed-shipped-but-wasn't.** web-dev shipped `cwdb_ios_submit_fix-1.2.0` (pure JS, calls `form.requestSubmit()` from `touchend` handler). Reported as "live in production." Subsequent diagnostic revealed it was registered but **never applied to the site** via `apply_to_site` call. The previous fix never took effect.

**Round 4 — Jim halted.** "Stop assuming. Also fails on mobile Chrome. Interview me before assuming." Entered plan mode for full rebuild. AskUserQuestion gathered actual failure mode (tap does literally nothing), confirmed all Playwright submissions had been arriving as emails (form pipeline OK; bug is in real-device click path), and field-state truth (name+email don't exist in form definition — they're JS-injected at runtime).

**Plan-mode discovery.** Webflow has NO native Multi-Step Form element. Webflow MCP cannot duplicate pages. Original plan's premise was partially wrong. Three options surfaced; Jim approved Option 1 (surgical fix on live page).

**Surgical execution:**
- Added `name` and `email` as real Webflow form fields on Step 2 (top of step)
- Moved `project_type` from Step 2 to Step 3 (first question)
- Renamed select-field `data-name` attributes to human-readable labels
- Built `cwdb_site_polish-2.0.1` consolidating: CTA glow CSS (without iOS-killing `inline-flex`), TCPA layout CSS, top-bar sticky banner JS, nav-rebuild fallback, TCPA wrapper for legacy pages
- Atomic swap: applied new polish + unapplied 5x `cwdb_launch_polish` (1.0.0–1.2.0) + 3x `cwdb_round_2_fixes` (1.0.0–1.2.0). Site script count: 15 → 8 applied.
- Deleted unapplied `cwdb_ios_submit_fix-1.2.0` from registry.

**Next button cleanup.** Jim flagged Next button text was upper-left aligned (not centered) on real device. Root cause: removing the flex centering rule fixed iOS submit but broke anchor-button text centering. Hit a separate MCP blocker: `data_scripts_tool` only exposes `add_inline_site_script` (~2000-char cap), can't update hosted `cwdb_site_polish-2.0.1` (4368 chars). Workaround: registered `quote_page_polish-1.0.0` via `upsert_page_script` — page-scoped to `/get-a-quote` only, doesn't count toward 15-slot site limit. Contains: `a.btn-submit { display: inline-flex; align-items: center; justify-content: center }` (anchor-only, iOS-safe) plus URL-param prefill for the homepage hero handoff.

**Production publish.** 2026-04-28 03:31 UTC to both `cwdeckbuilders.com` and `www.cwdeckbuilders.com` (required domain IDs not hostnames — initial attempt with hostnames returned 400). CDN propagation under 30 seconds.

**Real-device verification.** Jim confirmed pass on iPhone 11 Safari AND Chrome.

**Memory consolidation.** 5 new memory files capturing the structural lessons (see Memory Updates below).

## Decisions Made

1. **Surgical fix on live page over v2 page rebuild** — same architectural outcome (Webflow form + multistepwizard JS), 90 min vs 3-4 hrs, lower risk because we're editing what already submits successfully.
2. **Step layout reorganization** — Step 1: zip+phone (homepage hero handoff), Step 2: name+email+address+owns_property (contact + property), Step 3: project_type+budget+timeline+notes+TCPA (all project info colocated, per Jim's explicit request).
3. **Script consolidation strategy** — replace 8 stacked legacy versions with single `cwdb_site_polish-2.0.1`. Frees 7 slots; site count goes 15/15 → 8/15.
4. **Page-scoped scripts as the standard pattern** for /get-a-quote-only logic (`quote_page_polish-1.0.0`). Doesn't count toward site-script slot limit. Standard workaround for >2000-char inline cap when content can't go in `cwdb_site_polish` due to MCP char limit.
5. **Real-device testing is the only acceptance criterion** for mobile UI. Playwright Chromium passes are sanity checks only — they cannot reproduce iOS WebKit bugs because Chromium uses Blink and iOS Safari/Chrome both use WebKit per App Store rule.
6. **Anchor flex centering safe; submit-input flex centering banned.** WebKit's iOS bug only affects `<input type="submit">` form-control elements. `<a class="btn-submit">` anchors can use `display: inline-flex` freely. CSS rule MUST be split: input selector gets no flex; anchor selector gets flex.
