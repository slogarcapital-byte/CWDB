- **GA4 UI rejected SA email** at Property Access Management with "This email doesn't match a Google Account." Diagnosed as the `suiteusermanagement` endpoint calling Workspace identity validator instead of IAM. Account-level invite + incognito both failed with same error. Confirmed not a user-mistake or propagation issue.
- **Built `templates/scripts/grant-ga4-access.ps1`** — Admin API bypass using Jim's OAuth user token (reusing `GOOGLE_ADS_CLIENT_ID/SECRET` with fresh consent for `analytics.manage.users` scope). Pulls a one-shot access token, POSTs to `accessBindings`, throws away token.
- **Three iterations to get the grant working:**
  1. First 404 → diagnosed missing Analytics Admin API enablement (separate from Data API). Patched error-body capture using `curl.exe -D / -o / -w %{http_code}` instead of `Invoke-RestMethod`.
  2. After enable, still 404 → discovered accessBindings is **v1alpha** only, never promoted to v1beta. Changed URL.
  3. Then 400 INVALID_ARGUMENT with body `{ user: cwdb-ga4-reader...` (quotes stripped) → PowerShell 5.1's known bug stripping inner `"` chars during native-command arg passing. Fixed by writing body to temp file and using curl's `--data-binary @file` syntax.
- **SUCCESS:** `properties/533582902/accessBindings/AYVX2ewp7AUVCjYCnCuOdvIHhn_Wh2LhxJDo2Ald8d3KShFKeEJ4o2uCm91h-0Uwg4x4sRtVU4mH22MEELo1Ce2JSgviGE1v4s6Vsg==` created. SA `cwdb-ga4-reader@cwdb-ads-pull.iam.gserviceaccount.com` now has Viewer on property 533582902.
- **Jim downloaded JSON key, set `GA4_SERVICE_ACCOUNT_JSON=C:\Users\jslog\.cwdb\ga4-sa.json` in `.env.local`.**
- **`pull-ga4-7d.ps1` crashed:** `[RSACryptoServiceProvider] does not contain a method named 'ImportFromPem'`. Method was added in .NET 5; Windows PowerShell 5.1 runs on .NET Framework 4.x which doesn't have it. Required PowerShell 7.
- **PS 7 install adventure:** first winget invocation appeared to succeed but installed only a Microsoft Store stub at `WindowsApps\pwsh.exe` with no real binary. Second invocation said "already installed" but `Test-Path` confirmed no `pwsh.exe` anywhere on disk. `winget install --force` finally landed it as an Appx package at `C:\Program Files\WindowsApps\Microsoft.PowerShell_7.6.2.0_x64__8wekyb3d8bbwe`. Discovered via `Get-AppxPackage` since Appx installs don't appear in classic Add/Remove Programs registry keys.
- **Patched both `pull-ga4-7d.ps1` and `pull-ga4-warehouse.ps1`** with a self-relaunch shim at the top: detects PSVersion.Major < 6, finds pwsh via `Get-Command` or the Appx stub fallback, re-invokes self under pwsh with the same params. Cron config stays unchanged (still calls `powershell.exe`, shim handles re-launch transparently).
- **Ran `pull-ga4-7d.ps1` for real:** 7-day result = 111 sessions / 103 users / 0 conv, top channel Direct (52). Auth proven end-to-end.
- **Ran `pull-ga4-warehouse.ps1`:** 323 rows UPSERT'd into `fact_traffic_daily` covering 2026-04-18 → 2026-06-02 (46 days of GA4 history, since the site went live).

## Decisions Made

1. **Meta warehouse loader Get-PropOrNull fix shipped to main path.** Drop `[Parameter(Mandatory)]` from `$InputObject` so the function's existing null-guard runs. Affects future Meta pulls on days with no conversions — would have crashed every fresh-account day-1 load.
2. **Admin API bypass chosen over indefinite GA4 UI retries.** After A/B/C from the diagnostic checklist (identity check, account-level invite, incognito) all failed, ruled out user error / propagation / Workspace issues and went direct to backend. Saved ~hours of UI roulette.
3. **PowerShell 7 declared a hard requirement for GA4 scripts.** Alternative was ~80 lines of hand-rolled ASN.1 / PKCS#8 parsing in PS 5.1. Not worth the brittleness. PS 7 brings forward benefits for every future Google/AWS/HashiCorp script Jim runs.
4. **Self-relaunch shim over cron-config edits.** Both GA4 scripts auto-detect PS 5.1 and re-invoke under pwsh. Means cron (`powershell.exe -File <script>`) doesn't need updating, and Jim can invoke from any shell.
5. **Phase E deferred to a fresh session.** Phase D's data is now real ammunition for the views — building them sloppy now vs deliberately next session is the right tradeoff.
6. **Two data-quality findings carried into Phase E** as first-class view requirements: (a) `/thank-you = 27 sessions` vs `fact_leads` cross-check (potential lead-leak detector), (b) Meta-attributed-conversions-via-GA4 ≠ Meta-attributed-via-Meta (pixel routing issue surfaced by the warehouse, which was the entire point of building it).
