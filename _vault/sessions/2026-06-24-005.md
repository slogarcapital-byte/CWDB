# Session 2026-06-24-005

## Context
Jim reported the Mission Control dashboard showing `RUNNING · gate closed · stale_warehouse_data` and asked what needs to be updated. Goal: diagnose the stale-warehouse condition, fix it so the gate reopens, and (follow-up) explain what mechanism keeps the dashboard updated.

## Work Done
- **Diagnosed (not stale by age, a failing source).** The daily refresh was running, but the HubSpot pull failed (`exit=1`) on every run since 2026-06-24 06:20 Central while Google/Meta/GA4 all succeeded. `run-daily.ps1` marks the whole run failed if any source fails, and `Test-WarehouseFresh` (26h window, requires the last `RUN_END` to have `overall_exit=0`) then reports stale, which `control-tick` writes as `gate_reason=stale_warehouse_data`. Traced via `_vault/data/cron-runs.log` and `_vault/data/hubspot-error.log`.
- **Root cause.** A web-form test contact (`test@test.com`, HubSpot id `507581897445`, created 2026-06-23) collided with an orphaned `fact_leads` row (lead_id 7, from a 2026-05-21 test) on the `fact_leads_email_key` UNIQUE(email) constraint. The loader upserts on `hubspot_contact_id` (different IDs), so the new contact's INSERT 409'd on the separate email-unique constraint, failing the whole HubSpot batch.
- **Guard (the real fix).** Added a pull-level test-lead skip to `templates/scripts/pull-hubspot-snapshot.ps1`, mirroring `v_clean_leads` exactly (test emails, `@cwdb-internal.test` domain, `full_name` starting "test", `utm_source='test'`), applied BEFORE the `fact_leads` upsert. A view filter alone could not prevent this because the 409 happens at ingestion, before any view. Surfaced as `test_leads` in the skip summary.
- **Purge.** Deleted the orphaned `fact_leads` lead_id 7 via Supabase (verified 0 dependent `fact_bids` rows first). Could not delete the HubSpot contact (the HubSpot MCP has no delete/archive action and the pull token is read-only); the guard makes deletion unnecessary, so the contact stays inert and skipped at ingestion.
- **Re-ran + reopened the gate.** `run-daily.ps1` returned overall exit 0 (HubSpot now `test_leads=1` skipped, 11 leads upserted, all 4 sources clean). Ran `control-tick.ps1` to re-evaluate: `gate_open=True reason=open`. Confirmed persisted `control_state`: `run_mode=running, gate_open=true, gate_reason=open` with a fresh token.
- **Committed + pushed.** Commit `c050cd5` (only `pull-hubspot-snapshot.ps1`); pushed `origin/test-branch` and fast-forwarded `origin/main` (both at `c050cd5`).
- **Answered the dashboard-update question.** Three layers: (1) the port-aware babysitter keeps the server process alive; (2) the server reads Supabase live on every `/api/state` request; (3) Supabase state is kept current by `CWDB-Control-Tick` (every 30 min), `CWDB-Warehouse-Daily` (06:55), and the remote orchestrator routine (every 2h). KEY finding: the UI (`public/app.js`) does NOT auto-refresh. It fetches only on page load, on the manual Refresh button, and after an action ("Refresh-on-load only"). That is why Jim's open tab still showed the old stale banner after the data was fixed.
