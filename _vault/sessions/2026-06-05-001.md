The lock holder for the corrupted reparse was the kernel-level `cldflt.sys` filter driver. 39 files in the project had hardcoded OneDrive paths (vault-sync.ps1, agent .md files, settings.local.json), making full relocation costly.

### Junction surgery

1. `attrib +P /s /d` recursively pinned 820+ files. Verified pin attribute set (525361 = PINNED + REPARSE_POINT + DIRECTORY + ARCHIVE + READONLY).
2. First rmdir attempt failed with "Access denied" (cldflt filter holding lock even with READONLY cleared).
3. Stopped OneDrive process (PID 15384). Cleared READONLY attribute (`attrib -R`). rmdir still failed with "directory not empty" — cldflt was projecting 32 ghost files at the corrupted path.
4. `fsutil reparsepoint delete` failed with "Error 395: Access to the cloud file is denied" — cldflt requires admin to bypass.
5. **PowerShell `Remove-Item -Recurse -Force` succeeded** (treats the corrupted reparse contents as opaque files and deletes them as files first, then the empty dir).
6. `cmd /c mklink /J` created proper mount-point junction. Verified: `LinkType: Junction`, reparse tag `0xa0000003`, junction count == source count (41 == 41 at first, then 22 == 22 after later file moves).
7. Restarted OneDrive.
